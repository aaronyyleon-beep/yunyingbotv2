# 项目脚手架与代码结构设计说明

## 1. 文档目标

本文档用于定义 MVP 第一阶段建议采用的代码仓结构、后端模块落位方式、前后端目录划分、配置组织方式以及初始化顺序。

本文档解决的问题：

- 项目代码应该如何组织
- 分析层各模块应该落在哪些目录
- 前端和后端如何共存
- Prompt、因子配置、工作流配置放哪里
- 第一版仓库应该先搭哪些骨架

## 2. 当前判断

基于现阶段需求，我建议采用：

**单仓库 Monorepo + 后端优先 + 前端预留 + 配置驱动**

原因：

- 当前平台仍处于第一阶段，功能边界在快速收敛
- 分析层、工作流、Prompt、因子配置之间耦合较高
- 前后端都需要围绕同一套实体和配置迭代
- 用多仓库会明显增加联调和维护成本

因此第一版不建议拆成多个仓库。

## 3. 技术组织建议

### 3.1 顶层组织原则

建议把项目拆成四类内容：

- 应用层
- 领域层
- 配置层
- 文档层

### 3.2 第一版建议结构

建议仓库顶层结构如下：

```text
yunyingbotv2/
  apps/
  packages/
  configs/
  prompts/
  docs/
  scripts/
  data/
  infra/
```

说明：

- `apps`：放可运行应用
- `packages`：放可复用业务模块
- `configs`：放因子、维度、流程等配置
- `prompts`：放 AI 节点 Prompt 模板
- `docs`：放产品与架构文档
- `scripts`：放初始化、导入、开发辅助脚本
- `data`：放本地样例数据、fixtures、临时导入数据
- `infra`：放部署、环境、数据库迁移、容器等内容

## 4. 应用层建议

### 4.1 `apps/api`

用途：

- 后端主应用
- 提供任务接口、查询接口、复核接口、重算接口
- 承载后台业务 API

建议包含：

- Web API
- 任务调度入口
- 后台管理接口

### 4.2 `apps/worker`

用途：

- 承载异步工作流任务
- 执行收集、证据整理、因子分析、重算

建议包含：

- 全局扫描任务
- 定向补抓任务
- 因子分析任务
- 维度汇总任务
- 报告生成任务
- 重算任务

### 4.3 `apps/web`

用途：

- 前端应用
- 包含聊天输入端和分析工作台

说明：

- 现阶段先预留
- 真正 UI 开工时按 `frontend-design` 方案落地

## 5. 领域模块建议

建议把业务逻辑按领域拆到 `packages` 下，而不是全部堆进 `apps/api`。

### 5.1 建议目录结构

```text
packages/
  domain/
  application/
  integrations/
  shared/
```

### 5.2 `packages/domain`

用途：

- 放核心领域模型和规则

建议子目录：

```text
packages/domain/
  project/
  task/
  source/
  evidence/
  factor/
  dimension/
  report/
  review/
  version/
```

每个领域目录建议包含：

- entity / model
- repository interface
- enums / types
- domain rules

### 5.3 `packages/application`

用途：

- 放用例层和工作流层

建议子目录：

```text
packages/application/
  intake/
  identify-project/
  collect-global/
  collect-targeted/
  build-evidence/
  analyze-factors/
  aggregate-dimensions/
  generate-report/
  review-factor/
  recalculate/
```

说明：

- 这里对应我们前面拆出来的工作流阶段
- 每个目录代表一个主要 use case

### 5.4 `packages/integrations`

用途：

- 放外部集成能力

建议子目录：

```text
packages/integrations/
  llm/
  twitter/
  telegram/
  discord/
  web-parser/
  docs-parser/
  onchain/
  search/
  storage/
  database/
  queue/
```

说明：

- 所有外部调用都尽量从这里出入
- 避免业务层直接耦合第三方实现

### 5.5 `packages/shared`

用途：

- 放公共工具、基础类型、日志、错误、配置加载器等

建议子目录：

```text
packages/shared/
  types/
  utils/
  logger/
  errors/
  constants/
```

## 6. 配置层建议

这是这个项目很重要的一层，建议从第一版就独立。

### 6.1 `configs/factors`

用途：

- 存因子定义
- `factor_key`
- 因子描述
- 所属维度
- 输入证据类型要求
- 展示文案

### 6.2 `configs/dimensions`

用途：

- 存维度定义
- 维度与因子映射
- 汇总规则配置

### 6.3 `configs/workflows`

用途：

- 存工作流阶段配置
- 状态流转配置
- 收集停止规则

### 6.4 `configs/scoring`

用途：

- 存评分规则和统一阈值
- 第一版可先轻量实现

### 6.5 `configs/ui`

用途：

- 放前端状态标签、颜色语义、展示顺序等共享配置

## 7. Prompt 层建议

Prompt 不建议写死在业务代码里。

建议目录：

```text
prompts/
  identify-project/
  collect/
  analyze-factor/
  summarize-dimension/
  generate-report/
  review-assistant/
```

建议每类 Prompt 至少拆为：

- system prompt
- task template
- output schema example

说明：

- 后续迭代 AI 节点时会很频繁
- 独立目录比写在 service 里更可维护

## 8. 文档层建议

当前已有大量文档，建议统一整理到 `docs/` 下。

建议结构：

```text
docs/
  product/
  analysis/
  architecture/
  research/
  archive/
```

建议迁移方式：

- 产品与平台说明类放 `docs/product`
- 分析层、因子、数据库、工作流说明放 `docs/analysis`
- 架构图和后端结构说明放 `docs/architecture`
- 原始 docx / png 放 `docs/archive`

## 9. 数据与样例建议

建议增加 `data/` 目录，存放：

- 样例项目输入
- 样例证据数据
- 样例因子分析输出
- 假数据 fixtures

建议结构：

```text
data/
  samples/
  fixtures/
  imports/
```

这对后续调试工作流和前端联调很有帮助。

## 10. 基础脚本建议

建议在 `scripts/` 中准备以下脚本：

- 初始化本地开发环境
- 导入因子配置
- 导入维度配置
- 初始化测试数据
- 运行单任务分析流程
- 生成演示数据

## 11. 基础代码结构建议

后端建议采用“接口层 -> 应用层 -> 领域层 -> 集成层”的分层方式。

建议调用方向：

```text
API / Worker
  -> Application Use Cases
    -> Domain Rules
      -> Repository Interfaces
        -> Integrations / Database / External Services
```

这样可以避免：

- 控制器里写满业务逻辑
- 外部 API 调用散落各处
- AI Prompt 和领域规则混在一起

## 12. MVP 初始化顺序建议

建议按以下顺序起项目：

### 12.1 第一步：仓库初始化

- 建立顶层目录
- 建立包管理结构
- 建立基础 lint / format / env 规范

### 12.2 第二步：后端主应用骨架

- `apps/api`
- `apps/worker`
- 基础数据库连接
- 基础日志与配置系统

### 12.3 第三步：核心领域模型

- project
- task
- source
- evidence
- factor
- dimension
- report
- review
- version

### 12.4 第四步：核心工作流骨架

- intake
- identify-project
- collect-global
- build-evidence
- analyze-factors
- aggregate-dimensions
- generate-report

### 12.5 第五步：复核与重算

- review-factor
- recalculate
- versioning

### 12.6 第六步：前端预留与联调数据

- `apps/web` 初始化
- 接入假数据或演示接口

## 13. 技术栈建议

如果你没有强约束，我建议第一版优先选“开发效率高、类型清晰、适合工作流”的技术栈。

建议方向：

- 后端：TypeScript
- API：Node.js 框架
- Worker：同语言异步任务执行
- 数据库：PostgreSQL
- ORM / Query：类型友好的方案
- 前端：React

原因：

- 前后端可统一语言
- 因子配置、Prompt 配置、类型定义共享更方便
- 工作流和 AI 集成也更顺手

## 15. 当前结论

第一版代码结构最重要的不是“像大厂架构”，而是做到以下几点：

1. 前后端同仓，便于快速演进
2. 工作流阶段和代码模块保持一致
3. Prompt、因子、维度、规则独立于业务代码
4. 领域对象清晰，不把所有逻辑堆进接口层
5. 先保证分析层底座稳定，再逐步接前端和策略层

这套脚手架方案已经足够指导正式建仓和起第一版代码骨架。
