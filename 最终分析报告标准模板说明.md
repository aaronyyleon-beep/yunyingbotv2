# 最终分析报告标准模板说明

## 1. 文档定位

本说明用于定义分析层的最终报告输出标准。

该报告优先服务两类读者：

- 决策人
- 客户 / 项目方

报告风格要求为：

- 专业克制
- 判断明确
- 证据可解释
- 不情绪化
- 不销售化

本报告不替代分析工作台，也不替代策略层。
它的职责是将分析层的最终判断，以适合阅读和汇报的形式输出。

## 2. 报告目标

最终分析报告应回答以下问题：

- 当前项目整体处于什么状态
- 当前风险水平如何
- 哪些问题最影响后续判断
- 哪些信号仍值得保留
- 是否值得继续推进
- 如果继续推进，下一步应进入哪类策略讨论

## 3. 报告形态

系统内应同时存在两种报告形态。

### 3.1 工作台阅读版

用途：

- 内部复核
- 分析师下钻
- 查看因子、维度、证据、版本

特点：

- 强结构化
- 可跳转
- 可展开
- 可查看人工复核痕迹
- 可查看版本差异

### 3.2 导出 / 汇报版

用途：

- 给决策人阅读
- 给客户 / 项目方阅读
- 用于正式汇报或对外沟通

特点：

- 结论集中
- 语言克制
- 图表适量
- 证据摘要化
- 可独立阅读

## 4. 报告结构

最终分析报告采用固定六段式结构。

### 4.1 执行摘要

应包含：

- 项目名称
- 分析对象范围
- 一句话总体判断
- 综合风险等级
- 3 个最关键问题
- 1 到 2 个积极信号

不应包含：

- 过多技术细节
- 因子级逐条展开
- 大段背景介绍

### 4.2 综合判断

应包含：

- 当前项目总体状态
- 为什么得到该总体判断
- 哪些维度主导了当前判断
- 是否建议继续投入运营资源

不应包含：

- 冗长证据罗列
- 过度模糊的判断表述

### 4.3 维度概览

应包含：

- 各维度分数
- 各维度一句话判断
- 主要拖分维度
- 主要保留维度

不应包含：

- 全量因子展开
- 细碎原始证据

### 4.4 关键问题

应包含：

- 3 到 5 个最影响最终结论的问题
- 每个问题的业务含义
- 每个问题对后续运营判断的影响

不应包含：

- 所有小问题堆叠
- 只描述现象，不说明影响

### 4.5 关键证据

应包含：

- 官网 / 文档关键证据摘要
- Twitter 关键证据摘要
- Telegram / Discord 关键证据摘要
- 链上关键证据摘要

不应包含：

- 全量原始证据
- 大段聊天记录
- 原始 HTML 或原始长文本

### 4.6 结论与建议入口

应包含：

- 最终结论
- 是否建议继续推进
- 建议优先复核的区域
- 建议进入策略层前需要确认的条件

不应包含：

- 完整策略方案
- 完整执行计划

## 5. 数据来源映射

最终分析报告不需要新建独立数据体系，应基于现有分析层结果重组生成。

### 5.1 执行摘要

主要来源：

- `report.final_score`
- `report.risk_level`
- `report.summary`
- 低分维度
- 高风险因子

### 5.2 综合判断

主要来源：

- `report.summary`
- `dimensions`
- 高权重因子理由
- 最终确认版结果

### 5.3 维度概览

主要来源：

- `dimensions`
- 各维度主要因子

### 5.4 关键问题

主要来源：

- `factors`
- 低分因子
- 高风险因子
- 人工复核后仍保持低判断的因子

### 5.5 关键证据

主要来源：

- `evidences`
- `insight_summary`
- `twitter_detail`
- `community_detail`

### 5.6 结论与建议入口

主要来源：

- `report`
- `dimensions`
- `factors`
- `review_records`
- `versions`

正式导出的最终报告，默认使用最终确认版，而不是 AI 初版。

## 6. 写作原则

### 6.1 结论先于分数

先表达判断，再给出分数支持。

### 6.2 影响先于描述

不只描述发生了什么，还要说明这意味着什么。

### 6.3 证据先于口号

所有重要结论都应能追溯到证据摘要。

### 6.4 克制表达

避免夸张、情绪化或销售式表达。

### 6.5 风险优先

风险与机会都应写，但风险优先级高于机会。

## 7. 与策略层的边界

最终分析报告只负责完成分析层收口，不负责替代策略层。

报告最后一段只提供：

- 是否建议继续推进
- 进入策略层前的重点关注项
- 需要优先修正或复核的问题

不直接输出完整策略方案，不提前展开执行排期。

## 8. 当前系统落地建议

分析层当前阶段建议按以下顺序落地：

1. 保持工作台阅读版继续服务内部复核
2. 在现有 `report + dimensions + factors + evidences` 基础上生成正式汇报版
3. 汇报版默认绑定最终确认版
4. 关键证据采用摘要化输出，不直接暴露全量原始证据
5. 报告结尾保留策略层入口，不越界到策略方案正文

## 9. 结论

最终分析报告应定义为：

一份面向决策人和客户 / 项目方的、专业克制、证据可解释的分析输出。

它在系统中应同时存在：

- 工作台阅读版
- 导出 / 汇报版

前者服务复核与下钻，后者服务判断与沟通。
