# AI 规则分层与 Prompt 装载设计说明

## 1. 文档目标

本文档用于解决一个关键问题：

**随着平台规则、因子、流程和评分逻辑越来越多，如何避免 AI 上下文过长、判断变钝、输出僵化。**

本文档目标：

- 定义哪些规则应该长期常驻
- 定义哪些规则应该按阶段加载
- 定义哪些规则应该按因子加载
- 定义哪些逻辑不应交给 AI，而应由程序和配置负责

## 2. 核心结论

规则多本身不是问题。

真正的问题是：

- 把所有规则一次性塞给 AI
- 让 AI 同时记平台原则、全流程、全因子、全评分细则
- 让 AI 兼做判断、流程控制、规则计算和状态管理

这会带来 5 个典型问题：

- 上下文过长
- 输出变钝
- 忽略重点
- 解释僵硬
- 成本上升且稳定性下降

因此平台必须采用：

**规则分层 + Prompt 按需装载 + 程序主控流程**

## 3. 规则分层原则

建议把规则拆成 4 层。

### 3.1 L1：平台原则层

这是最少但最重要的一层。

作用：

- 约束 AI 的总体工作方式
- 定义平台不可违反的判断原则

特点：

- 数量很少
- 长期稳定
- 可常驻

建议内容：

- 证据先于结论
- 因子先于维度
- AI 主收集、主分析
- 人工可改单因子
- 输出必须可解释
- 对证据不足要诚实标注

这层适合做成系统级常驻 Prompt 或基础行为说明。

### 3.2 L2：流程阶段层

作用：

- 规定 AI 当前所处的工作阶段
- 约束 AI 只处理当前阶段任务

特点：

- 与当前工作流节点相关
- 不需要全流程同时出现

示例：

- 当前阶段是主体识别
- 当前阶段是全局扫描
- 当前阶段是定向补抓
- 当前阶段是单因子分析
- 当前阶段是维度汇总
- 当前阶段是复核辅助

这层应按阶段动态加载。

### 3.3 L3：任务对象层

作用：

- 告诉 AI 当前正在处理哪个具体对象

特点：

- 高度局部化
- 与当前任务强绑定

示例：

- 当前因子定义
- 当前维度目标
- 当前证据集合
- 当前用户补充说明
- 当前改分动作

这层必须按需加载，不应一次性装入全部对象。

### 3.4 L4：程序规则层

作用：

- 把不适合让 AI 记忆和自由发挥的逻辑，交给程序和配置执行

特点：

- 可计算
- 可枚举
- 需要稳定一致

示例：

- 状态流转
- 汇总计算
- 重算触发
- 版本生成
- 权限控制
- 阈值判断
- JSON Schema 校验

这层不应主要依赖 AI。

## 4. 哪些规则适合常驻

常驻内容必须非常克制。

建议只保留：

- 平台角色定义
- 证据优先原则
- 因子优先原则
- 诚实处理证据不足
- 输出格式总要求

常驻层不应包含：

- 全部因子清单
- 所有评分规则
- 所有阶段细节
- 所有维度定义
- 所有阈值表

原因：

- 这些内容变化频繁
- 数量庞大
- 当前任务通常只需要其中很小一部分

## 5. 哪些规则应按阶段装载

建议以下内容按阶段装载：

- 当前阶段目标
- 当前阶段输入说明
- 当前阶段允许输出什么
- 当前阶段禁止做什么
- 当前阶段需要关注的判断点

例如：

### 5.1 主体识别阶段

只需要：

- 如何识别主体
- 如何区分官网、Twitter、TG/DC、白皮书、合约
- 如何判断置信度

不需要：

- 全因子定义
- 全维度汇总逻辑
- 全报告模板

### 5.2 全局扫描阶段

只需要：

- 哪些来源优先
- 目标是建立来源地图和初始画像
- 不要求深挖所有细节

### 5.3 单因子分析阶段

只需要：

- 当前因子定义
- 当前证据
- 当前输出结构

不需要：

- 其他 19 个因子的定义

### 5.4 维度汇总阶段

只需要：

- 当前维度包含哪些因子
- 如何做维度摘要
- 输出哪些高风险点和高机会点

### 5.5 复核辅助阶段

只需要：

- 当前被改的是哪个因子
- 原始判断是什么
- 新补充的事实是什么
- 需要输出哪些影响说明

## 6. 哪些规则应按因子装载

这是最关键的一层。

每次 AI 分析单因子时，只应拿到当前因子的最小上下文：

- `factor_key`
- 因子定义
- 所属维度
- 当前证据列表
- 当前评分说明
- 当前输出格式

建议单因子 Prompt 结构固定化：

1. 当前因子是什么
2. 这个因子要回答什么问题
3. 当前有哪些证据
4. 证据不足时怎么处理
5. 输出哪些字段

这样做的好处：

- 上下文短
- 目标清晰
- 输出稳定
- 更容易测

## 7. 哪些逻辑不应交给 AI 记

这是最容易踩坑的地方。

以下内容建议尽量交给程序和配置，而不是让 AI“背下来”：

- 状态机流转
- 因子到维度的计算关系
- 汇总后的总分计算
- 风险等级映射
- 收集停止条件
- 重算触发条件
- 版本编号和版本生成
- 输出结构校验

原因：

- 这些内容需要稳定、可复现、不可漂移
- 交给 AI 会导致同样输入可能输出不同执行逻辑

AI 更适合做：

- 判断
- 解释
- 摘要
- 证据组织
- 风险和机会提炼

## 8. Prompt 结构建议

建议平台中的 Prompt 采用“基础层 + 阶段层 + 对象层”拼装，而不是单个超大 Prompt。

### 8.1 基础层

内容示例：

- 你是平台分析代理
- 必须遵守证据先于结论
- 必须如实表达证据不足
- 输出必须结构化

### 8.2 阶段层

内容示例：

- 你现在执行的是“单因子分析”
- 当前目标是判断某个因子
- 不要扩展到其他因子或总报告

### 8.3 对象层

内容示例：

- 当前因子定义
- 当前证据集
- 当前输出 schema

### 8.4 程序约束层

内容示例：

- 通过代码注入 schema
- 通过程序控制状态
- 通过程序做结果校验

这部分不一定是自然语言，也可以是系统调用约束。

## 9. 推荐的 Prompt 装载模式

### 9.1 主体识别 Prompt

装载：

- L1 平台原则
- L2 主体识别阶段说明
- 当前输入资料

### 9.2 全局扫描 Prompt

装载：

- L1 平台原则
- L2 全局扫描阶段说明
- 当前项目主体信息
- 当前已知来源

### 9.3 定向补抓 Prompt

装载：

- L1 平台原则
- L2 定向补抓阶段说明
- 当前缺口因子列表
- 当前可用来源清单

### 9.4 单因子分析 Prompt

装载：

- L1 平台原则
- L2 单因子分析阶段说明
- 当前因子定义
- 当前证据

### 9.5 维度汇总 Prompt

装载：

- L1 平台原则
- L2 维度汇总阶段说明
- 当前维度定义
- 当前维度下的因子结果

### 9.6 复核辅助 Prompt

装载：

- L1 平台原则
- L2 复核辅助阶段说明
- 当前因子原结果
- 人工补充事实
- 人工修正分

## 10. 控制 AI 变呆的具体方法

建议至少做以下 7 件事：

1. 常驻规则不超过一小屏
2. 单次 Prompt 只处理一个阶段目标
3. 单因子分析时不加载全因子库
4. 能用配置和代码做的不要交给 AI 记
5. 输出统一用结构化 schema 约束
6. 证据预处理后再给 AI，不把原始垃圾内容全喂进去
7. 让 AI 解释判断，不让 AI 负责整个平台状态控制

## 11. 文档与 Prompt 的关系

这里还有一个很重要的区分：

- 文档可以很长
- Prompt 必须很短

文档的作用是：

- 帮团队对齐
- 帮开发实现
- 帮定义规则边界

Prompt 的作用是：

- 在某一时刻完成某一个很具体的任务

所以不是把文档直接喂给 AI，而是把文档“编译”为：

- 固定系统规则
- 阶段模板
- 因子模板
- 配置与程序逻辑

## 12. 当前结论

要让 AI 在这个平台里保持聪明、稳定、不僵化，核心不是减少规则，而是正确拆规则：

1. 少量平台原则常驻
2. 工作流规则按阶段装载
3. 因子定义按对象装载
4. 稳定逻辑交给程序和配置
5. Prompt 始终保持短、小、专注

一句话总结：

**AI 不应该“背整本操作手册”，而应该在每一步只拿到当前工作卡片。**
