# 数据库设计说明书

## 1. 文档目标

本文档用于定义 MVP 第一阶段的核心数据模型、表结构方向、实体关系、状态流转和后续扩展约束。

本文档目标不是一次性设计完整生产库，而是先建立一套足够稳的分析层数据库骨架，支持以下能力：

- 项目与任务创建
- 来源映射
- 证据归档
- 单因子分析
- 维度汇总
- 总报告输出
- 人工改单因子
- 自动重算
- 版本留痕

## 2. 设计原则

### 2.1 任务驱动

分析是按任务运行的，因此绝大部分分析结果都应挂在 `analysis_task` 下，而不是直接挂在 `project` 下。

### 2.2 证据与结论分离

证据、因子、维度、报告、复核记录必须独立建模，避免后续无法追溯和重算。

### 2.3 原始值与最终值分离

AI 原始结果、人工修正结果、最终确认结果必须可同时保留。

### 2.4 支持版本化

分析结果不是只保留最新值，而是要支持快照和历史回看。

### 2.5 支持逐步扩展

第一版表结构不追求一步到位，但必须保证：

- 能新增因子
- 能新增维度
- 能新增来源类型
- 能新增证据类型
- 能新增策略层接口字段

## 3. 核心实体总览

MVP 第一阶段建议包含以下核心表：

- `projects`
- `analysis_tasks`
- `task_inputs`
- `sources`
- `evidences`
- `factors`
- `factor_evidence_refs`
- `dimensions`
- `reports`
- `review_records`
- `report_versions`

如需补充系统级能力，可后续增加：

- `users`
- `work_logs`
- `tool_runs`
- `prompt_runs`

## 4. 实体关系总览

建议关系如下：

- 一个 `project` 可以有多个 `analysis_task`
- 一个 `analysis_task` 可以有多个 `task_input`
- 一个 `project` 可以有多个 `source`
- 一个 `analysis_task` 可以有多条 `evidence`
- 一个 `analysis_task` 可以有多个 `factor`
- 一个 `factor` 可以关联多条 `evidence`
- 一个 `analysis_task` 可以有多个 `dimension`
- 一个 `analysis_task` 通常有一条当前 `report`
- 一个 `analysis_task` 可以有多条 `review_record`
- 一个 `analysis_task` 可以有多个 `report_version`

## 5. 表设计

### 5.1 `projects`

用途：

- 表示项目主体

建议字段：

- `id`
- `name`
- `display_name`
- `aliases_json`
- `primary_chain`
- `official_website`
- `official_twitter`
- `official_discord`
- `official_telegram`
- `contract_addresses_json`
- `status`
- `created_at`
- `updated_at`

说明：

- `project` 是长期对象
- 主体识别成功后应优先复用已有 `project`

### 5.2 `analysis_tasks`

用途：

- 表示一次完整分析任务

建议字段：

- `id`
- `project_id`
- `task_name`
- `task_status`
- `collection_status`
- `analysis_status`
- `review_status`
- `final_status`
- `current_version_id`
- `started_at`
- `finished_at`
- `created_by`
- `created_at`
- `updated_at`

说明：

- 核心流程状态建议都挂在任务上
- `current_version_id` 指向当前主版本

### 5.3 `task_inputs`

用途：

- 保存任务原始输入

建议字段：

- `id`
- `task_id`
- `input_type`
- `raw_value`
- `normalized_value`
- `source_label`
- `uploaded_by`
- `created_at`

说明：

- 用于追踪用户最初到底提交了什么
- 便于后续回溯任务上下文

### 5.4 `sources`

用途：

- 保存项目的外部来源映射

建议字段：

- `id`
- `project_id`
- `source_type`
- `source_url`
- `source_name`
- `is_official`
- `source_owner`
- `access_method`
- `access_status`
- `metadata_json`
- `first_seen_task_id`
- `created_at`
- `updated_at`

说明：

- `source` 偏“入口级对象”
- 一个来源可被多个任务复用

### 5.5 `evidences`

用途：

- 保存任务中抓取到的标准化证据

建议字段：

- `id`
- `task_id`
- `source_id`
- `evidence_type`
- `title`
- `raw_content`
- `normalized_content`
- `summary`
- `credibility_level`
- `is_official_source`
- `captured_at`
- `parser_name`
- `metadata_json`
- `created_at`

说明：

- 证据必须挂在任务上
- 同一来源在不同任务中可以产出不同证据

### 5.6 `factors`

用途：

- 保存单因子分析结果

建议字段：

- `id`
- `task_id`
- `factor_key`
- `factor_name`
- `dimension_key`
- `dimension_name`
- `status`
- `ai_score`
- `human_override_score`
- `final_score`
- `confidence_level`
- `score_reason`
- `risk_points_json`
- `opportunity_points_json`
- `fact_supplement`
- `override_reason`
- `reviewed_by`
- `reviewed_at`
- `created_at`
- `updated_at`

说明：

- `factor_key` 应为稳定键，便于规则和代码引用
- `final_score` 为当前任务视角下的最终值

### 5.7 `factor_evidence_refs`

用途：

- 保存因子与证据的多对多关系

建议字段：

- `id`
- `factor_id`
- `evidence_id`
- `relation_type`
- `weight`
- `created_at`

说明：

- `relation_type` 可区分：
  - `primary_support`
  - `secondary_support`
  - `risk_support`
  - `opportunity_support`

### 5.8 `dimensions`

用途：

- 保存维度汇总结果

建议字段：

- `id`
- `task_id`
- `dimension_key`
- `dimension_name`
- `ai_score`
- `final_score`
- `summary`
- `positive_factors_json`
- `negative_factors_json`
- `high_risk_points_json`
- `high_opportunity_points_json`
- `chart_payload_json`
- `created_at`
- `updated_at`

说明：

- 维度由因子聚合得出，不建议人工直接改维度分

### 5.9 `reports`

用途：

- 保存任务当前主报告

建议字段：

- `id`
- `task_id`
- `final_score`
- `risk_level`
- `operational_mode_suggestion`
- `summary`
- `core_risks_json`
- `core_opportunities_json`
- `chart_payload_json`
- `report_body_json`
- `created_at`
- `updated_at`

说明：

- `reports` 偏“当前态”
- 历史态放到 `report_versions`

### 5.10 `review_records`

用途：

- 保存每次人工复核动作

建议字段：

- `id`
- `task_id`
- `factor_id`
- `reviewer`
- `old_ai_score`
- `old_final_score`
- `override_score`
- `new_final_score`
- `fact_supplement`
- `override_reason`
- `impact_summary`
- `created_at`

说明：

- 每次人工改动都必须留痕

### 5.11 `report_versions`

用途：

- 保存任务版本快照

建议字段：

- `id`
- `task_id`
- `version_type`
- `version_label`
- `based_on_version_id`
- `factor_snapshot_json`
- `dimension_snapshot_json`
- `report_snapshot_json`
- `change_summary_json`
- `created_by`
- `created_at`

说明：

- 推荐支持：
  - `ai_initial`
  - `human_revised`
  - `final_confirmed`

## 7. JSON 字段使用建议

为加快 MVP 开发，部分高变动字段可以先使用 JSON 存储。

建议可先 JSON 化的字段：

- `aliases_json`
- `contract_addresses_json`
- `metadata_json`
- `risk_points_json`
- `opportunity_points_json`
- `positive_factors_json`
- `negative_factors_json`
- `high_risk_points_json`
- `high_opportunity_points_json`
- `chart_payload_json`
- `report_body_json`
- `change_summary_json`

原则：

- 高频查询字段尽量结构化
- 高变动、展示型字段可先 JSON 化

## 8. 快照与当前态并存策略

建议同时保留：

- 当前态表：`factors`、`dimensions`、`reports`
- 快照表：`report_versions`

这样可以兼顾：

- 当前页面快速读取
- 历史版本完整回放

建议逻辑：

- 页面默认读取当前态
- 生成版本时，把当前态序列化写入快照

## 9. 重算数据更新建议

当单因子发生修改时，建议按以下顺序更新：

1. 更新 `factors`
2. 写入 `review_records`
3. 重算 `dimensions`
4. 重算 `reports`
5. 生成 `report_versions`
6. 更新 `analysis_tasks.current_version_id`

这样可以保证当前态和历史态一致。

## 10. 索引建议

MVP 阶段建议至少建立以下索引：

- `analysis_tasks.project_id`
- `task_inputs.task_id`
- `sources.project_id`
- `evidences.task_id`
- `evidences.source_id`
- `factors.task_id`
- `factors.factor_key`
- `factor_evidence_refs.factor_id`
- `factor_evidence_refs.evidence_id`
- `dimensions.task_id`
- `reports.task_id`
- `review_records.task_id`
- `review_records.factor_id`
- `report_versions.task_id`

## 11. 数据约束建议

### 11.1 主约束

- 一个任务只能有一个当前主报告
- 一个任务可以有多个版本快照
- 一个因子只能属于一个维度
- 一个复核记录必须对应一个因子

### 11.2 业务约束

- 人工不能直接改 `dimensions.final_score`
- 人工不能直接改 `reports.final_score`
- 所有汇总结果必须通过重算产生

## 12. 后续扩展预留

数据库设计应预留以下扩展方向：

- 接入 `users` 与权限系统
- 接入 `tool_runs`，记录每次外部抓取和工具调用
- 接入 `prompt_runs`，记录 AI 推理输入输出
- 接入 `strategy_drafts`，承接策略层
- 接入 `execution_plans`，承接执行层

## 13. 当前结论

MVP 第一阶段的数据库设计重点，不是做一个完美大而全的库，而是先保证以下四件事：

1. 项目、任务、来源、证据、因子之间关系清楚
2. AI 原始结果、人工修正结果、最终结果能并存
3. 改单因子后能稳定重算
4. 所有关键结果都能留痕和回放

这套结构一旦稳定，后面的工作流、接口层和前端工作台都会顺很多。
